# 外部信号原料池第一阶段实施计划

日期：2026-04-29

## 目标

第一阶段交付一个文件式外部原料池工具，让人、agent 和调度任务都能把外部材料写进同一个池子，也能稳定查询、追溯和反馈。

本阶段只解决原料问题。机会判断、摘要、评分、后台、数据库和推荐逻辑全部延后。

## 完成标准

完成后必须能跑通这条闭环。

人投一个链接，系统生成批次、标准化材料、原始引用或明确失败记录。

agent 提交一组现场采集材料，材料进入同一套格式。

调度或手动任务能跑至少一个稳定渠道并写入批次。

agent 能按渠道、时间、关键词、类型、采集者和反馈状态查材料。

材料能追溯到来源链接、批次和原始文件。

材料或材料包能追加 useful、not_useful、adopted 三类反馈。

所有目标渠道都有测试样本，测试运行能记录成功、失败、字段缺失和受限样本。

## 实施顺序

### 阶段 0：确定工程边界

先选定第一版代码仓库、运行语言和 CLI 入口。后端同学可以决定具体技术栈，但必须保留文件协议、追加写入、失败显式记录这三条约束。

代码结构建议按职责拆开：schema、storage、adapters、cli、scheduler、fixtures、tests。目录名可以按实际技术栈调整，但这些边界不要混在一起。

### 阶段 1：文件池和数据协议

实现 `init-pool`，创建 `tasks`、`batches`、`materials`、`raw`、`bundles`、`feedback`、`tests`、`test-runs` 目录。

定义并校验这些 JSON/JSONL 结构：batch manifest、material record、error record、task definition、bundle record、feedback record、fixture record、test-run record。

写入必须是追加式。批次不能覆盖，原材料不能被反馈或补采改写。字段缺失和失败要进入错误记录或批次备注，不能用默认值假装正常。

### 阶段 2：CLI 最小闭环

实现第一批 CLI 命令：`submit-batch`、`query-materials`、`feedback`、`create-bundle`。

`submit-batch` 接收 agent 产出的 JSON 或 JSONL，校验最小字段，生成批次，写入标准化材料和 raw 引用。

`query-materials` 默认返回精简结果，支持按渠道、时间、关键词、材料类型、采集者、批次、反馈状态筛选。展开模式再返回正文、关联材料和 raw 引用。

`feedback` 只追加 useful、not_useful、adopted，不修改原材料。`adopted` 支持带产出链接。

`create-bundle` 只保存材料 ID、标题、创建者、创建时间、组包原因和可选标签，不写总结或判断。

### 阶段 3：人工投递

实现 `submit-link`，支持单链接和多链接投递。

链接先路由到渠道适配器。没有适配器时走通用网页兜底。兜底也要生成正常批次，不能因为抓不全就丢掉投递记录。

抓取成功时保存标准化材料和 raw 引用。抓取失败时保存原链接、失败类型、失败原因和批次记录。

这一阶段先用 YouTube、B站、GitHub、Steam 或普通网页做端到端样本，证明投递链路可用。

### 阶段 4：适配器框架

实现统一适配器契约。每个适配器至少要能识别支持的 URL 或任务目标，返回标准化材料，保存 raw，记录错误，跑自己的 fixture。

适配器支持成熟度分级。Level 0 能接收并记录失败，Level 1 能抓核心字段，Level 2 能抓评论和互动，Level 3 能抓历史变化和补采。

所有渠道都创建适配器占位和 fixture 目录。即使某个渠道暂时只有 Level 0，也要把能力边界写清楚。

### 阶段 5：首批可跑渠道

优先打通结构稳定、容易验收的渠道，建议先做 GitHub、Reddit、YouTube、Steam、App Store、Google Play 中的三到四个。

每个可跑渠道至少支持一种任务或链接投递方式，能产出核心字段、raw 引用和错误记录。

X/Twitter、TikTok、Instagram、小红书、抖音、微博、B站、Discord、微信小游戏也要纳入适配器和测试集，但抓取深度可以逐步补。

### 阶段 6：任务和轻量调度

实现 `run-task`，读取任务定义，执行指定适配器，生成新批次。

实现轻量调度，只做按计划触发任务、记录开始结束时间、保存成功和失败结果。不做智能重试、机会分析、摘要或渠道优化。

同一个任务重复执行时，每次都生成新批次，不覆盖历史。

### 阶段 7：全渠道测试集

为每个目标渠道建立 fixture。每个渠道至少覆盖普通内容、带评论内容、高互动内容、低互动内容和边界内容。

边界内容包括已删除、需要登录、地区受限、字段缺失、异常链接、空结果任务或页面结构变化。

实现 `run-tests`，输出 test-run 结果。测试通过不要求所有样本都成功，但要求成功样本产出合法材料，失败样本产出明确失败记录。

### 阶段 8：验收和交接

用真实样本跑完整链路。至少覆盖人工投递、agent 批量写入、任务执行、查询、反馈、材料包、测试集运行。

整理一份后端交接说明，说明文件协议、CLI 命令、适配器契约、fixture 格式、失败记录规则和延后事项。

验收时重点看数据是否真实存在、是否能追溯、失败是否可见、agent 是否好消费。不要用页面、摘要或机会判断当验收标准。

## 测试策略

协议层测试校验 JSON/JSONL 结构、必填字段、追加写入、缺失字段错误、非法反馈类型、重复批次 ID。

CLI 测试覆盖初始化、导入 agent batch、查询、反馈、材料包、人工链接投递、任务执行。

适配器测试用 fixture 跑全渠道样本，分别检查成功材料和失败记录。

回归测试要覆盖几个不能妥协的规则：失败不能变空结果，原材料不能被覆盖，反馈不能改原材料，agent 写入不能绕过统一 schema。

## 风险和处理

最大风险是社媒渠道抓取不稳定。处理方式不是暂停系统，而是把这些渠道先做成 Level 0 或 Level 1，允许人工投递和 agent 现场采集先进入池子。

第二个风险是大家各自写不同格式。处理方式是先做 schema 校验和 CLI 入口，让系统采集、人投递、agent 写入都走同一套协议。

第三个风险是测试样本太少，适配器看起来可用但真实链接一跑就坏。处理方式是每个渠道都要有多样本 fixture，并把失败样本也视为测试资产。

第四个风险是早期把分析能力塞进材料池。处理方式是守住边界，材料池只保存事实和采用反馈，不写机会判断。

## 第一版里程碑

M1 完成文件协议、init-pool、submit-batch、query-materials、feedback。

M2 完成人工 submit-link、raw 保存、错误记录和普通网页兜底。

M3 完成适配器框架和三到四个可跑渠道。

M4 完成 run-task、轻量调度和全渠道 fixture。

M5 跑通端到端验收，交给后端同学继续加深渠道和服务化。

## 延后事项

数据库、全文检索、对象存储、向量索引、API 服务、后台、摘要、评分、机会库、渠道权重优化都不进入第一阶段。

这些能力以后可以基于同一套材料、批次、任务、材料包和反馈记录扩展，不能反过来污染第一阶段的原始材料协议。
