# Antigravity Provider Migration Spec

## Purpose

The project should treat Antigravity as the provider identity, not Gemini. The current code is halfway migrated: user-facing labels mostly say Antigravity, but the durable provider id, scripts, environment keys, function names, tests, and some command aliases still use `gemini`.

This spec defines the migration from primary provider id `gemini` to primary provider id `antigravity`, while preserving backward compatibility for existing state, service names, command habits, and environment files.

## Current Scan

Scan command used:

```bash
rg -n --glob '!node_modules' --glob '!package-lock.json' '(gemini|Gemini|GEMINI)' .
```

High-signal remaining areas:

| Area | Evidence | Migration intent |
| --- | --- | --- |

| Runner and event plumbing | `runner-args.js`, `runner-event-handlers.js`, `runner-executor.js` import and branch on `gemini` | Canonical branches should use `antigravity`; old aliases normalize into it |

Current count by file shows the largest hotspots are `src/provider-sessions.js`, `test/report-formatters.test.mjs`, `test/provider-sessions.test.mjs`, `test/provider-metadata.test.mjs`, `test/runtime-bootstrap.test.mjs`, README files, and runtime env tests.

## Migration Principles

The canonical provider id becomes `antigravity`.

`gemini`, `google`, and `agy` remain accepted input aliases. They should normalize to `antigravity`, not to `gemini`.

Existing persisted data must not be orphaned. Any channel, thread, provider bucket, dedicated bot state file, process lock, and service label using `gemini` must continue to work.

Compatibility names must be explicit and visible in code as legacy support. They should not keep the project mentally anchored to Gemini.

No fallback should hide malformed state. Missing legacy data is fine. Corrupt state files, malformed provider buckets, or conflicting migration data must be reported clearly and must not be silently overwritten.

## Target Behavior

New configs use `ANTIGRAVITY__*`.

New direct binary config uses `ANTIGRAVITY_BIN`.

New locked provider mode uses `BOT_PROVIDER=antigravity`.

New npm scripts include `npm run start:antigravity` and `npm run dev:antigravity`.

Old entry points still work:

```bash
npm run start:gemini
BOT_PROVIDER=gemini npm start
GEMINI__DISCORD_TOKEN=...
GEMINI__DEFAULT_WORKSPACE_DIR=...
GEMINI_BIN=...
scripts/restart-discord-bot-service.sh gemini
```

When both new and legacy env keys exist, the new Antigravity key wins. The legacy key should still be reported in diagnostics as present but shadowed.

The provider picker and help should show `antigravity`, not `gemini`.

`!provider gemini` and slash values generated by old clients should still resolve to Antigravity, but fresh command surfaces should not advertise Gemini as the provider.

Session terminology should be `conversation`, `conversations`, `conversation_resume`, and `conversation_sessions`. Existing `chat_resume` and `chat_sessions` should keep working as aliases for at least one compatibility window.

## Data Migration

### Session Records

Session records may contain:

```json
{
  "provider": "gemini",
  "providers": {
    "gemini": {
      "runnerSessionId": "...",
      "codexThreadId": "...",
      "model": "..."
    }
  }
}
```

After migration, loaded state should become:

```json
{
  "provider": "antigravity",
  "providers": {
    "antigravity": {
      "runnerSessionId": "...",
      "codexThreadId": "...",
      "model": "..."
    }
  }
}
```

If both `providers.gemini` and `providers.antigravity` exist, merge rules must be deterministic:

`antigravity` wins for fields that are explicitly non-empty. Empty/null fields in `antigravity` may be filled from `gemini`. Conflicting non-empty fields must be logged or surfaced as a migration warning instead of silently choosing one if the field affects session binding, workspace, or model.

The migration must be idempotent. Running it twice must not duplicate provider buckets, change session ids, or flip sources.

### State Files

Dedicated files may exist as:

```text
data/sessions.gemini.json
data/channel-runtime.gemini.json
data/project-upgrade.gemini.json
```

Canonical files should become:

```text
data/sessions.antigravity.json
data/channel-runtime.antigravity.json
data/project-upgrade.antigravity.json
```

If the new file is missing and the legacy file exists, load the legacy file, migrate it in memory, then write the canonical file. Do not delete the legacy file during this migration. Deletion can be a later cleanup step.

If both files exist, use the canonical file. Do not merge whole files automatically. Report the presence of the legacy file as a compatibility notice.

### Workspace Binding

Antigravity remains workspace-bound. Existing code that resets sessions on workspace changes must keep that behavior.

The legacy `resolveGeminiProjectRootBySessionId` should become `resolveAntigravityProjectRootBySessionId`. Internally it still needs to read two sources:

Antigravity cache at `~/.gemini/antigravity-cli/cache/last_conversations.json`.

Legacy Gemini CLI project data at `~/.gemini/tmp` and `~/.gemini/projects.json`.

The `.gemini` directory name is owned by Google tooling and must not be renamed in code comments or paths.

## Code Migration Plan

### Provider Metadata

Change `PROVIDER_METADATA.gemini` to `PROVIDER_METADATA.antigravity`.

Set aliases to:

```js
['antigravity', 'agy', 'gemini', 'google']
```

`normalizeProvider('gemini')` should return `antigravity`.

`parseOptionalProvider('gemini')` should return `antigravity`.

Provider display name remains `Antigravity CLI`.

Provider short name remains `Antigravity`.

Default bin remains `agy`.

Default env key becomes `ANTIGRAVITY_BIN`.

Default slash prefix needs a product decision. The clean new default is `ag`, but changing this can break existing dedicated bot command habits. Recommended migration is: new canonical locked provider defaults to `ag`; legacy `start:gemini` continues to default to `gm`; explicit `GEMINI__SLASH_PREFIX` still works.

### Provider Adapter

Rename:

```text
src/providers/gemini.js -> src/providers/antigravity.js
createGeminiProviderAdapter -> createAntigravityProviderAdapter
```

Keep a small compatibility module at `src/providers/gemini.js` only if imports outside the repo or staged work still depend on it. It should re-export the Antigravity adapter and be marked legacy.

### Runtime Args and Executor

Rename Antigravity-specific functions and injected dependencies:

```text
buildGeminiArgs -> buildAntigravityArgs
readGeminiSessionState -> readAntigravitySessionState
resolveGeminiProjectRootBySessionId -> resolveAntigravityProjectRootBySessionId
geminiDeltaBuffer -> antigravityDeltaBuffer
GEMINI_BIN -> ANTIGRAVITY_BIN
geminiBin -> antigravityBin
```

The command shape remains:

```bash
agy --sandbox --conversation <id> --prompt <text>
agy --dangerously-skip-permissions --conversation <id> --prompt <text>
```

No `--model` flag should be added unless `agy --help` starts exposing one.

### Runtime Env

Add Antigravity scoped env support:

```text
ANTIGRAVITY__DISCORD_TOKEN
ANTIGRAVITY__DEFAULT_MODE
ANTIGRAVITY__DEFAULT_WORKSPACE_DIR
ANTIGRAVITY__CHILD_THREAD_WORKSPACE_MODE
ANTIGRAVITY__SLASH_PREFIX
ANTIGRAVITY__ANTIGRAVITY_BIN
ANTIGRAVITY_BIN
DISCORD_TOKEN_ANTIGRAVITY
```

Legacy fallbacks remain:

```text
GEMINI__*
GEMINI_BIN
DISCORD_TOKEN_GEMINI
```

Precedence:

```text
ANTIGRAVITY__KEY > GEMINI__KEY > unscoped KEY
ANTIGRAVITY_BIN > GEMINI_BIN > agy
DISCORD_TOKEN_ANTIGRAVITY > DISCORD_TOKEN_GEMINI > DISCORD_TOKEN
```

Diagnostics should show when a legacy key is used.

### Package Scripts and Service Scripts

Add:

```json
"start:antigravity": "node scripts/start-instance.mjs antigravity",
"dev:antigravity": "BOT_PROVIDER=antigravity node --watch src/index.js"
```

Keep:

```json
"start:gemini": "node scripts/start-instance.mjs gemini",
"dev:gemini": "BOT_PROVIDER=gemini node --watch src/index.js"
```

`scripts/start-instance.mjs` should accept `antigravity` and `gemini`, with `gemini` normalized to `antigravity`.

`scripts/restart-discord-bot-service.sh` should accept `antigravity`, `gemini`, and both old/new launchctl labels.

`src/launchctl-guard.js` and `scripts/guard-bin/launchctl` should allow and recommend the new Antigravity label while still protecting old Gemini labels.

### Commands and UI

Provider choices should show `antigravity`.

Text help should say:

```text
!provider <codex|claude|antigravity|status>
```

Old `!provider gemini` should still work.

Add native aliases:

```text
conversation_sessions
conversation_resume
```

Keep old aliases:

```text
chat_sessions
chat_resume
```

Descriptions should use `conversation`, not `chat session`, for Antigravity.

The provider settings panel should use `antigravity` as the selected value and custom id value. Legacy custom ids containing `gemini` should be parsed if an old interaction is still live.

### Docs

README should prefer:

```bash
npm run start:antigravity
scripts/restart-discord-bot-service.sh antigravity
```

`.env.example` should prefer `ANTIGRAVITY__*`.

A compatibility note should say that `GEMINI__*`, `start:gemini`, and `gm_` were the old Gemini-provider names and still work during migration.

Sample workspace paths should not use `gemini-work`.

Historical changelog entries can keep Gemini.

### Tests

Canonical tests should use `antigravity`.

Legacy compatibility tests should use `gemini`.

Any test that says `Gemini CLI` should be removed unless it is intentionally testing legacy Gemini CLI session files.

Model names such as `Gemini 3.5 Flash (High)` are valid Antigravity model labels and should not be renamed.

## Adversarial Test Spec

### Static Scans

Run after migration:

```bash
rg -n --glob '!node_modules' --glob '!package-lock.json' 'Gemini CLI|gemini CLI|Gemini provider|provider key.*gemini' README.md README.en.md docs src test .env.example
```

Expected: no current product-facing Gemini provider language.

Run:

```bash
rg -n --glob '!node_modules' --glob '!package-lock.json' '(provider === .gemini.|provider: .gemini.|BOT_PROVIDER=gemini|start:gemini|GEMINI__)' src test README.md README.en.md docs .env.example package.json scripts
```

Expected: remaining hits are only legacy compatibility code, compatibility tests, compatibility docs, or `.gemini` Google-owned local cache paths. Every remaining code hit should be intentionally named `legacyGemini` or be covered by a test.

Run:

```bash
find src test docs scripts -iname '*gemini*'
```

Expected: either no canonical source files, or only explicit legacy compatibility shims such as `legacy-gemini-*`.

### Unit Tests

Provider normalization:

```js
normalizeProvider('antigravity') === 'antigravity'
normalizeProvider('agy') === 'antigravity'
normalizeProvider('gemini') === 'antigravity'
parseOptionalProvider('google') === 'antigravity'
```

Provider metadata:

```js
getProviderDisplayName('antigravity') === 'Antigravity CLI'
getProviderDefaultBin('antigravity') === 'agy'
getProviderBinEnvName('antigravity') === 'ANTIGRAVITY_BIN'
```

Env precedence:

```js
ANTIGRAVITY__DISCORD_TOKEN wins over GEMINI__DISCORD_TOKEN
DISCORD_TOKEN_ANTIGRAVITY wins over DISCORD_TOKEN_GEMINI
ANTIGRAVITY_BIN wins over GEMINI_BIN
```

Script mode parsing:

```text
node scripts/start-instance.mjs antigravity
node scripts/start-instance.mjs gemini
```

Both must start the Antigravity provider path. The legacy command should not create a separate Gemini state identity.

Session migration:

Use temp files with old state:

```json
{
  "provider": "gemini",
  "providers": {
    "gemini": {
      "runnerSessionId": "agy-conv-1",
      "model": "Claude Opus 4.6 (Thinking)"
    }
  }

```

Expected loaded session:

```json

  "provider": "antigravity",
  "providers": {
    "antigravity": {
      "runnerSessionId": "agy-conv-1",
      "model": "Claude Opus 4.6 (Thinking)"
    }
  }

```

Legacy dedicated state file:

```text

data/sessions.antigravity.json missing
```

Expected: Antigravity locked bot loads the legacy file and writes the canonical file without deleting the legacy file.

Conflict state:

```json

  "provider": "gemini",
  "providers": {
    "gemini": { "runnerSessionId": "old" },
    "antigravity": { "runnerSessionId": "new" }
  }
}
```

Expected: `new` wins, `old` is not silently copied over it, and a migration warning is observable in logs or report output.

Malformed state:

```text

```

Expected: no empty replacement file is written. Startup fails or reports a clear load error.

Commands:

```text
!provider antigravity
!provider gemini
!conversation_sessions
!conversation_resume <id>
!chat_sessions
!chat_resume <id>
```

Expected: all route to the same Antigravity provider. Fresh help advertises Antigravity and conversation commands; old chat commands still work.

Runner args:

Antigravity safe mode with session id:

```text
agy --sandbox --conversation <id> --prompt <text>
```

Antigravity dangerous mode:

```text
agy --dangerously-skip-permissions --conversation <id> --prompt <text>
```

Expected: no `gemini` binary, no `gemini` CLI args, no `--model` unless the CLI supports it.

### Regression Suite

Run the full project suite:

```bash
npm run test:progress
```

The suite must pass after the migration. Add targeted tests before migration code lands; do not rely only on broad snapshots.

## Rollout Steps

First land metadata normalization and tests so `antigravity` becomes canonical while `gemini` still resolves.

Then land env and script aliases so users can start Antigravity using the new name before state migration.

Then land session-store migration and dedicated file suffix handling.

Then rename adapter/functions and update imports.

Then update command surfaces and docs.

Then run adversarial verification and full regression.

## Non-Goals

Do not rename Google-owned local paths such as `~/.gemini/antigravity-cli`.

Do not remove legacy Gemini CLI session reading in the same migration. It is still needed to avoid breaking old stored sessions.

Do not implement Antigravity hot runtime in this migration. That requires a separate PTY or SDK design.

Do not add an Antigravity `--model` runner argument unless `agy --help` exposes one.

## Completion Criteria

New users see Antigravity everywhere.

Existing users with `gemini` state and env keep working.

All active runtime behavior uses the same canonical provider identity.

No data file is silently discarded or overwritten during migration.

Static scans have only allowlisted legacy Gemini references.

Adversarial verification checks and `npm run test:progress` pass.
